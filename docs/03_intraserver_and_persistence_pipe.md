# The Intraserver & Persistence Pipe
### How State Moves Within the Backend, and How the GodHead Endures

> Governs the backend's internal circulation and its persistent store. Inherits the Central Dogma; the store *is* the single source of truth every law depends upon.

## 0. Charge

Specify how state persists and how it moves between backend components once data is inside the system. This document defines the substrate the entire order stands on: if persistence is unsound, nothing above it means anything. Author it gaplessly. Where mechanism is left open, it is marked — but the *guarantees* are not negotiable.

## 1. The Store — Substrate

**1.1 — Single abstracted store, one interface.** All backend components speak to the store through **one abstraction layer**, never to the database directly. The concrete backend may change (hosted Postgres now, self-hosted later, local for offline use) via configuration, not rewrite.

**1.2 — The substrate is PostgreSQL with `pgvector`.** Relational state and embedding vectors live in **one database**. Vector similarity search runs natively via `pgvector`. This consolidation is deliberate: one store to run, back up, and reason about — no second system to synchronize.

**1.3 — Raw bytes live on disk, by reference.** File content (the immutable atoms) is stored on the filesystem; the database holds **references** to those bytes plus all metadata. The database never swallows raw blobs.

**1.4 — The atom/bond separation is physical, not just conceptual.** File content (atoms) is immutable once written. Links between files (bonds) are mutable relational records. These are stored **separately**: mutating a link never touches a file. The volatility of the link layer is quarantined from the stability of the content layer.

## 2. The Data Model

Specify and enforce the following as rigidly standardized, versioned schemas.

**2.1 — Nodes (files).** The atoms. Raw content by reference; normalized-derivative content by reference; metadata (name, type, size, timestamps, normalized-state). Content is immutable once committed.

**2.2 — Embeddings.** One persisted vector per node, in `pgvector`. Never recomputed when it can be read. Generated by a local embedder (see the ML pipe). Similarity between embeddings is the measurable basis of all linkage.

**2.3 — Links (edges) — first-class objects.** The bonds. Each is its own record, never written into file bodies: `source · target · similarity/weight · category · user_overridden(bool) · schema_version · timestamps · provenance-ref`. This table *is* the graph.

**2.4 — Matrices.** Emergent clusters, recorded once link-density under a shared category crosses the **coherence threshold** — a single system-wide constant, never a private or local value. A matrix references its constituent nodes and links; its status is tracked explicitly (`POSTULANT → CARDINAL | DISSOLVED` — Dogma VI, A.9). `[Fable: the coherence threshold value is to be determined empirically.]`

**2.5 — Provenance records.** For any created datum: what produced it, from what inputs, when, under what schema version. Environment provenance-context (which agent fetched a thing, its tier, the driving brief, any follow-ups) is structured, queryable state — not free text.

**2.6 — Logs.** Append-only, rotating: every significant change writes a snapshot; new snapshots rotate prior ones into redundancy; nothing overwrites.

**2.7 — Readiness flags & job records.** The state that drives orchestration (§3): standardized flags certifying stage-completion-with-integrity, and records of agent invocations sufficient to support idempotency and recovery.

## 3. Intraserver Circulation

**3.1 — Handoffs pass through the store.** Components hand state to one another by way of the store rather than talking to each other directly — a stage writes validated output and a readiness flag, and the next stage reads them. This is the practical realization of the Handoff Law, and it is what keeps the pipeline decoupled and crash-resilient: state always lives on disk, never only inside a running component.

**3.2 — Hybrid orchestration: flag-driven transitions + an observing supervisor.**
- **Flag-driven transitions.** A thin dispatcher watches for readiness flags and invokes the next job. Intelligence lives in the state, not the dispatcher.
- **The supervisor.** A lightweight, server-resident bookkeeping process (not a hypervisor, not separate compute) reads flags to maintain a live index of every file's current stage — answering "where is everything?" It observes and may **nudge** (retry stalls, surface stuck jobs) but does not *drive* the pipeline.

**3.3 — Synchronization barriers.** Some transitions require *multiple* prerequisites, not one — the principal case being the two Auditors, who must *each* complete before a subsequent reconvene-and-propose step may run. The supervisor owns these AND-barriers as **certifier, not driver**: when all required completion flags are present and their underlying state validates, it writes a composite readiness flag; the dispatcher — as for every transition — performs the invocation. *(Barrier mechanics resolved: Dogma Book II §2, audit protocol.)*

**3.4 — Idempotency & recovery.** Because agents may be retried after a crash before termination, job execution must be idempotent-safe: a re-run must not duplicate or corrupt state. Job records (§2.7) support detecting and safely resuming interrupted work. *(Mechanism resolved: job-keyed idempotent writes, leases with heartbeat and expiry, compare-and-swap revisions — Dogma Laws I.3 and XI.)*

**3.5 — The Deacon's threshold sits here.** No externally-fetched material enters the store except through the Deacon's mandatory scan and human-authorization gate. Intake circulation must route returning-Student material through the Deacon before any write to persistent state.

## 4. Endurance Guarantees

**4.1 — The store survives restarts, crashes, and agent deaths.** Because agents hold no state and the store holds all of it, recovery is a property of the store alone. On restart, the supervisor reconstructs its live index from persisted flags and job records; no in-flight knowledge is lost because none lived only in an agent.

**4.2 — Raw-copied-once is inviolable.** Raw content is copied into persistence exactly once, at intake; that copy is the atom and is never rewritten by any downstream process.

**4.3 — Derivative-to-source integrity.** Every normalized derivative carries a persistent, unbreakable reference to the exact raw file it was produced from. Because raw is preserved untouched and copied only once, any derivative is disposable and regenerable: if normalization is later found faulty (for example, a mis-detected source encoding producing corrupted UTF-8), the derivative can be discarded and re-derived from the preserved raw. No committed data depends on a derivative being correct on the first pass. The raw file is the recoverable ground truth; the derivative is always reproducible from it.

**4.4 — Commitment is durable and logged.** A committed matrix is persisted as committed; reversal is human-invoked only and itself logged.

**4.5 — Forward-looking: multi-tenancy.** The intended future includes a small trusted group (home-server users), not only a single operator. The persistence layer must **not foreclose** eventually distinguishing whose data is whose, or who committed what. This need not be built now, but single-user assumptions that would be painful to unwind later must be avoided. `[Fable: note, do not over-build.]`

## 5. Deferred

**5.1 — The Deacon's cleanup pass.** A periodic, hardcoded, matrix-agnostic housekeeping routine (rotate what must rotate, prune orphaned nodes, tidy the commons), owned solely by the Deacon. Implemented only after the foundation stands. Ownership is settled; scheduling and safe-operation bounds are for later.

## 6. Standing Charge

Guarantee persistence and circulation gaplessly. The store is the ground truth on which every other document depends; pressure-test it against crash, retry, and partial failure. Leave open only what is marked open.
